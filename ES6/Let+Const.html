    块级作用域的出现

    # 块级作用域的出现

    通过 var 声明的变量存在变量提升的特性：

    if (conditioin) {
	var value = 1;
}
console.log(value);
    1
    2
    3
    4

    初学者可能会觉得只有 condition 为 true 时，才会创建 value，如果 condition 为 false，结果应该报错，然而因为变量 提升的原因，代码相当于：

    var value;
if (condition) {
	value = 1;
}
console.log(condition);
    1
    2
    3
    4
    5

    如果 condition 为 false，结果会是 undefined。

    除此之外，在 for 循环中：

    for( var i = 0; i < 10; i++) {
	...
}
console.log(i); // 10
    1
    2
    3
    4

    即使循环已经结束了，我们依然可以访问 i 的值。

    为了加强对变量生命周期的控制，ECMAScript6 引入了块级作用域。

    块级作用域存在于：

    • 函数内部
    • 块中（字符{和}之间的区域）

    # let 和 const

    块级声明用于声明在指定作用域外无法访问的变量。

    let 和 const 都是块级声明的一种。

    我们来回顾一下 let 和 const 的特点：

    1. 不会被提升
    if (false) {
	let value = 1;
}

console.log(value); // Uncaught ReferenceError: value is not defined
    1
    2
    3
    4
    5
    1. 重复声明报错
    let value = 1;
let value = 2; // Uncaught SyntaxError: Identifier 'value' has already been declared
    1
    2
    1. 不绑定全局作用域

    当在全局作用域中使用 var 声明的时候，会创建一个新的全局变量作为全局对象的属性。

    var value = 1;
console.log(window.value); // 1
    1
    2

    然而 let 和 const 不会：

    let value = 1;
console.log(window.value); // undefined
    1
    2

    再来说下 let 和 const 的区别：

    const 用于声明常量，其值一旦被设定就不能再修改，否则会报错。

    值得一提的是：const 声明不允许修改绑定，但允许修改值，这意味着当用 const 声明对象时：

    const data = {
	value: 1
}

// 没有问题
data.value = 2;
data.num = 3;

// 报错
data = {}; // Uncaught TypeError: Assignment to constant variable.

    # 临时死区

    临时死区（Temporal Dead Zone），简写为 TDZ。

    let 和 const 声明的变量不会被提升到作用域顶部，如果在声明之前访问这些变量，会导致报错：

    console.log(typeof value); // Uncaught RefrenceError: value is not defined
let value = 1;
    1
    2

    这是因为 JavaScript 引擎在扫描代码发现变量声明时，要么将其提升到作用域顶部（遇到 var 声明），要么将其放在TDZ中（遇到 let 和 const声明）。访问 TDZ 中的变量会触发运行时错误。只有当执行过变量声明语句后，变量才会从 TDZ 中移出，然后才可以访问。

    看似很好理解，不保证不会犯错：

    var value = "global";

// 例子1
(function() {
	console.log(value);

	let value = "local";
}());

// 例子2
{
	console.log(value);
	let value = "local";
}

    8

    10
    11
    12
    13
    14

    两个例子中，结果并不会打印 "global"，而是报错 Uncaught ReferenceError: value is not defined，就是因为 TDZ 的缘故。

    var funcs = [];
for (var i = 0; i < 3; i++) {
	funcs[i] = function () {

    # 循环中的 let 和 const

    不过到这里还没有完，如果我们把 let 换成 const 呢？

    var funcs = [];
for (const i = 0; i < 3; i++) {
    funcs[i] = function () {
        console.log(i);
    }
}
funcs[0](); // Uncaught TypeError: Assignment to constant variable
    1
    2
    3
    4
    5
    6
    7

    结果会是报错，因为虽然我们每次都创建了一个新的变量，然而我们却在迭代中尝试修改 const 的值，所以最终会导致报错。

    说完了普通的 for 循环，那么还有 for in 循环呢~

    那下面的结果是什么呢？

    var funcs = [], object = {a: 1, b: 1, c: 1};
for (var key in object) {
    funcs.push(function(){
        console.log(key)
    });
}

funcs[0]();
    1
    2
    3
    4
    5
    6
    7
    8

    结果是 "c"；

    那如果把 var 改成 let 或者 const 呢？

    使用 let，结果自然是正确值 "a"，这是因为在 for in 循环中，每次迭代不会修改已有的值，而是会创建一个新的绑定。

    另外，const 也能得出正确值而不会报错。这是与普通 for 循环不一样的地方。

    # 最佳实践

    在我们开发的时候，可能认为应该默认使用 let 而不是 var ，这种情况下，对于需要写保护的变量要使用 const。然而另一种做法日益普及：默认使用 const，只有当确实需要改变变量的值的时候才使用 let。这是因为大部分的变量的值在初始化后不应再改变，而预料之外的变量之的改变是很多 bug 的源头。
